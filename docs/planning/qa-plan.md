> Version: 1.0 | Status: In Force | Date: 2026-04-13
> Covers: Pre-work, Phases 0 through 5 (Phase 6 deferred per plan)

## 1. Test Architecture

SanitAI's test suite is organized into five distinct layers. Each layer has a
specific job; no layer duplicates the job of another.

```
┌─────────────────────────────────────────────────────────────┐
│  Layer             Tool                   CI Gate           │
├─────────────────────────────────────────────────────────────┤
│  Unit              cargo test (per crate) Must pass         │
│  Integration       cargo test --test '*'  Must pass         │
│  Regression corpus tools/differential     P≥0.98, R≥0.95   │
│  Context eval      tools/context-eval     P≥0.90, R≥0.85   │
│  Fuzz              cargo fuzz             No crash          │
│  Security lint     tools/security-lint/   Zero matches      │
│  Benchmark         cargo bench            No regression     │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 Unit tests

Location: `#[cfg(test)]` blocks inside each module (`regex_detector.rs`,
`transform.rs`, `cross_turn.rs`, `keyword_filter.rs`, etc.).

Scope: pure-function correctness, validator gate behavior, edge cases the
caller controls completely. Examples: `shannon_entropy`, `luhn_valid`,
`iban_valid`, `KeywordFilter::scan`, `ContextClassifier::classify` against
constructed inputs.

Anti-pattern to avoid: do not write unit tests that mock the `RegexDetector`
and then assert it was called. Test observable behavior, not call graphs.

### 1.2 Integration tests

Location: `crates/<crate>/tests/*.rs` (separate compiled test binaries).

Scope: crate-level behavior exercised from the outside, using real
implementations at all layers below the boundary. The `sanitai-cli`
integration tests invoke the compiled binary over `Command`. The
`sanitai-store` integration tests open a real SQLite file. Parser integration
tests feed real-format fixture files through the parser and assert on the
extracted `Turn` slice.

The `#[ignore]` pattern already in use for binary-dependent tests is correct.
Keep it. CI runs `cargo test -- --ignored` as a second step after
`cargo build`.

### 1.3 Regression corpus

Location: `tools/differential` (already exists). Corpus JSONL lives in
`crates/sanitai-fixtures/corpus/` (one file per secret class, generated by
`tools/corpus-gen`).

Scope: precision and recall of the detection engine across all rule classes.
Every new rule added in Phase 1a/1b must be accompanied by corpus entries
before the PR merges. The differential tool already enforces the gates:
precision >= 0.98, recall >= 0.95.

### 1.4 Context eval

Location: `tools/context-eval` (to be created in Phase 5). Labeled corpus at
`corpora/context/index.jsonl`.

Scope: precision and recall of the `ContextClassifier` specifically on the
`RealPaste` label. This is separate from the regression corpus because it
measures a different axis: not whether a regex fires, but whether the
contextual classification is correct.

### 1.5 Fuzz targets

Location: `fuzz/` at workspace root, using `cargo-fuzz` (libFuzzer).

Scope: parser, transform cascade, keyword filter, SARIF serializer. Fuzz
targets run in CI for 60 seconds per target on every PR touching the relevant
crate. Crash = any panic, OOM, or stack overflow. Acceptable: `Err` return,
empty output, truncated parse.

### 1.6 Security lint

Location: `tools/security-lint/check_no_log_secrets.sh` (to be created in
Phase 0.7).

Scope: `git grep` across all `.rs` files for patterns that would pass
`matched_raw` or `expose_secret()` to a `tracing` macro. Runs as a
pre-build step in CI. A match is a hard build failure, not a warning.

### 1.7 Benchmarks

Location: `crates/sanitai-detectors/benches/` using `criterion`.

Scope: keyword filter throughput and full-scan throughput on a 1MB synthetic
corpus. Not a pass/fail gate today — tracked as a time series to catch
regressions introduced by Phase 2+ hot-path changes.

### 1.8 Property-based tests

Location: `crates/sanitai-core/tests/properties.rs` (already exists with
properties 4-7 active, 1-3 stubbed pending real `scan()`).

Scope: scanner-wide invariants that must hold for all inputs — no panics,
spans within bounds, spans on char boundaries, findings do not overlap. These
properties must be un-ignored immediately when `scan()` is wired up.

### Phase 1a — Safe Rule Expansion

**Done condition:** all rules have passing positive and negative unit tests,
differential corpus gates still green.

For every rule added in Phase 1a, the following three test cases must exist:

**Template — one row per rule:**

| ID | Test name pattern | Input | Pass condition |
|---|---|---|---|
| `1a-<rule>-pos` | `detects_<rule_id>` | A minimal string containing exactly one valid instance of the secret | Exactly one finding with the correct `detector_id` |
| `1a-<rule>-neg-fmt` | `<rule_id>_rejects_short_prefix` | The prefix alone (e.g., `"discord"` without the token body) | Zero findings with this `detector_id` |
| `1a-<rule>-neg-corpus` | `<rule_id>_corpus_negative` | A corpus-negative entry: plausible text that contains the keyword but no token | Zero findings with this `detector_id` |

Apply this template to every rule in the Phase 1a table. Specific rules with
additional requirements:

| Rule | Additional test required |
|---|---|
| `discord_bot_token` | Negative: a Discord guild ID (snowflake) must not match the bot token pattern |
| `sendgrid_api_key` | Negative: `"SG."` as a sentence fragment must not match |
| `twilio_account_sid` | Negative: a generic 34-character hex string without `AC` prefix must not match |
| `gitlab_pat_variants` | 13 positive tests — one per prefix (`glpat-`, `gldt-`, ...) |
| `huggingface_token` | Negative: `"hf_"` with only 5 chars after must not match (length gate) |
| `fly_io_api_token` | Positive: must include the full `FlyV1 ` prefix + 100+ base64 chars |

**Corpus entries required per rule (added to `tools/differential` corpus):**

Each rule must have at minimum:
- 5 positive entries: realistic surrounding text, varied source (Claude export,
  ChatGPT export, raw paste format)
- 2 negative entries: text containing the keyword but not the full token pattern,
  and text from a different context entirely

Location: inline unit tests in `crates/sanitai-detectors/src/regex_detector.rs`;
corpus entries in `crates/sanitai-fixtures/corpus/<rule_id>.jsonl`.

## 3. Security Invariant Tests

The no-log-secrets invariant is the most critical correctness property of
SanitAI. A tool that scans for secrets must not itself leak them.

### 3.1 Mechanical enforcement via secrecy::Secret

The `matched_raw` field on `Finding` must be `secrecy::Secret<String>` after
Phase 0 ships. The `secrecy` crate's `Secret<T>` type:
- Implements `Debug` as `[REDACTED]`
- Implements `Display` as `[REDACTED]`
- Requires explicit `.expose_secret()` to access the inner value
- Implements `ZeroizeOnDrop` via the `zeroize` feature

Tests P0-01 through P0-04 verify this at the unit level. Test
`scan_json_output_never_contains_matched_raw` (already committed in
`integration.rs`) verifies it at the binary level.

### 3.2 access site restriction

After Phase 0, `.expose_secret()` must appear in exactly two crates:
`sanitai-core` (construction) and `sanitai-store` (hashing and DB write).
No other crate may call `.expose_secret()`.

Enforcement: add to CI as a `git grep` check:

```bash
#!/usr/bin/env bash
# Fail if expose_secret() is called outside permitted crates.
VIOLATIONS=$(git grep -n 'expose_secret' -- '*.rs' \
  | grep -v '^crates/sanitai-core/' \
  | grep -v '^crates/sanitai-store/')
if [ -n "$VIOLATIONS" ]; then
  echo "FAIL: expose_secret() called outside permitted crates:"
  echo "$VIOLATIONS"
  exit 1
fi
```

This is a stronger guarantee than a code review comment.

### 3.3 tracing macro check (Phase 0.7)

The existing `tools/security-lint/check_no_log_secrets.sh` script pattern
catches `tracing::*` macros that receive `matched_raw` or `expose_secret`.

The script should also check for the `log` crate as a defense-in-depth
measure, since a dependency could pull it in:

```bash
if git grep -n 'log::.*matched_raw\|log::.*expose_secret' -- '*.rs'; then
  echo "FAIL: secret value passed to log macro"
  exit 1
fi
```

### 3.4 VerifiedStatus::Error no raw HTTP strings

When Phase 6 is eventually implemented, add a compile-time test:

```rust
// In sanitai-verifier (Phase 6 only)
fn _no_raw_http_error_string_in_error_variant() {
    // This must not compile:
    // VerifiedStatus::Error("HTTP 401 Unauthorized: invalid token sk-ant-...")
    // The Error variant must use VerifyErrorKind, not String.
    // Verified by the type system — this test is documentation only.
}
```

### 3.5 ZeroizeOnDrop for cross-turn buffer

`TurnTail` already derives `ZeroizeOnDrop`. The test for this is implicit —
`zeroize` itself is tested by its own crate. The obligation is:

| ID | Test name | What it exercises | Pass condition |
|---|---|---|---|
| SEC-01 | `turn_tail_zeroize_on_drop` | Create a `TurnTail` with known content, drop it, verify the backing memory is zeroed | Use `ManuallyDrop` + raw pointer inspection to confirm bytes are 0 after drop |

This test is in `crates/sanitai-detectors/src/cross_turn.rs`. It is unsafe
code in a test context and must be gated behind `#[cfg(test)]` with a clear
safety comment.

## 4. Corpus Strategy (Phase 5)

### 4.1 Corpus location and structure

```
corpora/context/
  index.jsonl          # labeled entries, one JSON object per line
  schema.json          # JSON Schema for index.jsonl entries
  README.md            # labeling guidelines and contribution process
  splits/
    train.jsonl        # 80% for classifier development
    eval.jsonl         # 20% held out, never used to tune heuristics
```

The `eval.jsonl` split is locked. It is never modified once created. It is the
official number that CI gates on. The `train.jsonl` split is what implementers
run during development.

### 4.2 Entry schema

```json
{
  "id": "ctx-0001",
  "expected_class": "RealPaste",
  "source": "synthetic|claude_export|chatgpt_export|cursor|copilot|gemini",
  "turns": [
    { "role": "user|assistant|system|tool", "content": "..." }
  ],
  "finding_turn_idx": 0,
  "finding_byte_range": [12, 52],
  "detector_id": "anthropic_api_key",
  "notes": "user pastes .env file content verbatim"
}
```

The `id` field must be stable (never reassigned) to make CI diffs readable.

### 4.3 Label classes and minimum counts (post-baseline)

| Class | Minimum entries | Guidance |
|---|---|---|
| `RealPaste` | 200 | User-role turns, no surrounding educational prose, realistic context (`.env` paste, config paste, support request) |
| `Educational` | 150 | Assistant explains key format; code examples with placeholders; tutorial walkthroughs |
| `DocumentationQuote` | 75 | Surrounding text references docs URLs; finding appears in inline code or quote block |
| `ModelHallucination` | 75 | Known hallucination signatures: `AKIAIOSFODNN7EXAMPLE`, repeated-char patterns, `ghp_xxxx...` |

Minimum 500 total. The `RealPaste` class is weighted higher because it is the
class we must not miss (false negatives on real secrets are the failure mode
that matters).

### 4.4 Contribution and review process

1. New entries are submitted as PRs to `corpora/context/`.
2. Each PR must include a `notes` field for every entry explaining the labeling
   rationale.
3. At least one reviewer must independently verify the label for entries in
   `RealPaste` and `ModelHallucination` (the two highest-consequence classes).
4. The `eval.jsonl` split is append-only. Existing entries may not be
   re-labeled without a formal justification commit message and two approvals.
5. Synthetic entries (generated by `tools/corpus-gen`) do not require manual
   review. Human-curated entries (from real exports, anonymized) require review.

### 4.5 How CI enforces gates

`tools/context-eval` is added to the CI pipeline as a step that runs AFTER
`cargo build` and AFTER `cargo test`. It produces two observable outputs:

1. Exit code: non-zero if precision on `RealPaste` < 0.90 or recall on
   `RealPaste` < 0.85 against `eval.jsonl`. Blocks merge.
2. Artifact: `results/context_eval_report.md` uploaded to CI. Visible in PR
   checks without needing to run locally.

The tool uses `eval.jsonl` only. It must never read `train.jsonl`. If the wrong
split is used, the gate is meaningless.

### 4.6 Stale corpus detection

Add a CI check that fails if more than 90 days have passed since any entry was
added to `index.jsonl`. A stale corpus means the Phase 5 classifier is being
evaluated against the same distribution it was tuned on forever.

This is implemented as a script that inspects `git log --follow -- corpora/context/index.jsonl`
and compares the most recent commit date to today.

## 5. Fuzz Target List

Fuzz targets live in `fuzz/fuzz_targets/`. Each target is a separate binary
compiled with AddressSanitizer. Run via:

```bash
cargo fuzz run <target> -- -max_total_time=60 -max_len=65536
```

### 5.1 Target: fuzz_chatgpt_parser

**Input:** arbitrary bytes treated as a potential ChatGPT export JSON file.

**Feed point:** the byte slice is passed directly to
`sanitai_parsers::chatgpt::parse(input: &[u8])`.

**Crash condition:** any `panic!`, stack overflow, heap-use-after-free
(AddressSanitizer), or OOM.

**Acceptable outcomes:** `Err(...)` of any kind, `Ok(vec![])`, or `Ok(turns)`
where turns contain valid UTF-8 content.

**Corpus seeds:** `crates/sanitai-cli/tests/fixtures/chatgpt_single_turn.json`
(once created), plus 5 hand-crafted minimal valid inputs and 3
hand-crafted malformed inputs (truncated, wrong field types, deeply nested DAG).

### 5.2 Target: fuzz_cursor_parser

**Input:** arbitrary bytes treated as a SQLite file.

**Feed point:** bytes are written to a temp file, then
`sanitai_parsers::cursor::parse(path: &Path)` is called.

**Crash condition:** same as above.

**Special concern:** SQLite's built-in parser is robust but `rusqlite` with
`bundled` may have edge cases on malformed headers. The fuzz target must
confirm graceful `Err` on the magic byte check failure.

**Corpus seeds:** minimal valid SQLite (16-byte header + one page), the fixture
`cursor_state.vscdb` (once created), 3 corrupt variants.

### 5.3 Target: fuzz_transform_cascade

**Input:** arbitrary bytes treated as turn content fed to `TransformDetector`.

**Feed point:** bytes are wrapped in a `Chunk` and passed to
`TransformDetector::scan()` with a tiny budget (`max_decode_bytes: 1024`,
`max_depth: 2`).

**Crash condition:** panic, OOM exceeding 256MB (checked via AddressSanitizer
`--allocator-may-return-null`), or stack overflow.

**Acceptable outcomes:** any number of findings (including zero), any `Ok`.

**What this must not do:** exceed the decode budget. The test asserts
`ctx.decoded_total <= config.max_decode_bytes` via an instrumentable wrapper
or by checking the warning was emitted.

**Corpus seeds:** known base64 payload (from `transform.rs` test), known
gzip-within-base64 (double-encoded), 100-byte random high-entropy blob.

**Critical coverage target:** the gzip decompression bomb path. A 20-byte
gzip input that expands to 10MB must be absorbed without OOM. Include this
as a permanent seed.

### 5.4 Target: fuzz_keyword_filter

**Input:** arbitrary UTF-8 string as haystack, fixed rule set.

**Feed point:** `KeywordFilter::scan(haystack: &str)` on the production rule
set.

**Crash condition:** panic, any bit set outside the range `[0, rule_count)` in
the returned bitmask.

**Acceptable outcome:** any bitmask where only bits < rule_count are set.

**Invariant under test:** the returned `[u64; 4]` must satisfy
`bitmask[i] & (!valid_mask_for_bucket_i) == 0` for all buckets `i`.

### 5.5 Target: fuzz_sarif_serializer

**Input:** a randomly generated `Finding` (using proptest-style generation
seeded from the fuzz input bytes).

**Feed point:** `sarif::serialize(findings: &[Finding]) -> String`.

**Crash condition:** panic, invalid UTF-8 in output.

**Acceptable outcome:** a non-empty string that is valid JSON (checked by
`serde_json::from_str`). The fuzz target asserts `serde_json::from_str(&output).is_ok()`.

**Why this matters:** SARIF serialization could construct invalid JSON if any
field value contains unescaped special characters (e.g., a secret containing
a backslash or double-quote). The fuzz target exercises this.

## 6. Regression Prevention

### 6.1 The principle

Every test written for Phase N must continue to pass when Phase N+1 ships.
The CI pipeline runs the full test suite on every PR, so this is structurally
guaranteed — but only if tests are not incorrectly marked `#[ignore]` or
conditionally compiled away.

Rules:
- Never add `#[ignore]` to a previously-passing test.
- Never add `#[cfg(feature = "...")]` to an existing test to "silence it"
  during a refactor. Fix the test.
- If a Phase N behavior genuinely changes in Phase N+1 (rare), the test must
  be updated, and the change must be explicitly documented in the PR description
  with the old behavior and new behavior stated.

### 6.2 Pre-work / Phase 0 baseline lock

Before any Phase 1a code is merged, the `tools/differential` corpus must be
run and its output committed to `results/baseline_pre_phase1a.md`. This creates
a concrete reference point. If Phase 1b later causes precision to drop on an
existing rule, the regression is immediately visible in the diff.

### 6.3 Rule provenance as regression anchor

Every rule added in Phase 1a has a source comment:

```rust
// source: gitleaks/rules/discord.toml @ abc1234
```

If gitleaks updates that rule's pattern, the source comment makes the diff
visible in code review. This prevents silent divergence where our pattern
drifts from the battle-tested original.

### 6.4 Corpus entries as the regression corpus

Every new rule must be accompanied by corpus entries in
`crates/sanitai-fixtures/corpus/<rule_id>.jsonl` before the PR merges.
`tools/differential` is run as part of CI on every PR. A new rule that
reduces precision on an existing rule (e.g., because the new rule's pattern
overlaps with an existing rule's negative cases) will cause a CI failure.

This is the regression prevention mechanism for detection quality. It is more
valuable than code coverage because it catches actual behavioral regressions,
not just code path coverage.

### 6.5 Migration regression

Every schema migration must be tested both on a fresh DB and on a DB from the
previous version. The integration test P0-09 (`migration_v2_on_existing_v1_db`)
must be re-run as a template for any future migration (v3, v4, etc.). The
pattern is: create a SQLite file programmatically at schema version N-1 using
raw SQL, then open it with `Store::open_at`, verify old data is intact and
new columns exist with NULL values.

### 6.6 Phase 5 classifier drift detection

As new corpus entries are added, the precision/recall may move. The CI gate
enforces minimum bounds but does not flag improvement. A separate weekly CI
run produces a trend report (`results/context_eval_history.jsonl`) appended with
each run's precision/recall and a timestamp. If precision drops more than 5
percentage points in a single PR, a soft warning is emitted (non-blocking but
visible in CI summary). This is implemented as:

```bash
prev=$(jq -r '.[-1].precision' results/context_eval_history.jsonl)
curr=$(./target/release/context-eval --format json | jq '.precision')
drop=$(echo "$prev - $curr" | bc)
if (( $(echo "$drop > 0.05" | bc -l) )); then
  echo "WARNING: classifier precision dropped by $drop since last run"
fi
```

## 8. CI Pipeline Order

The following steps run in order on every PR. A failure at any step stops the
pipeline:

```
1. tools/security-lint/check_no_log_secrets.sh   (pre-build, ~1s)
2. tools/security-lint/check_expose_secret_scope.sh  (pre-build, ~1s)
3. cargo fmt --check                              (~2s)
4. cargo clippy -- -D warnings                   (~30s)
5. cargo build --workspace                       (~60s)
6. cargo test --workspace                        (~30s)
7. cargo test --workspace -- --ignored           (~15s, requires binary)
8. tools/differential --corpus <corpus>          (~60s)
9. tools/context-eval --corpus corpora/context/splits/eval.jsonl  (Phase 5+)
10. cargo fuzz run fuzz_transform_cascade -- -max_total_time=60   (~60s)
11. cargo fuzz run fuzz_chatgpt_parser    -- -max_total_time=60   (Phase 3+)
12. cargo fuzz run fuzz_keyword_filter    -- -max_total_time=60   (Phase 2+)
13. cargo fuzz run fuzz_sarif_serializer  -- -max_total_time=60   (Phase 4a+)
14. cargo bench -- --no-run  (compile check; full bench is nightly-only)
```

Steps 9-13 are added as their respective phases ship. Steps 1-8 are active
from Phase 0 onward.

The fuzz steps run for 60 seconds each in CI as a regression safeguard. They
are not intended to find new bugs in CI — that is the job of dedicated fuzz
campaign runs (manual, or on a schedule). The 60-second runs replay the
existing corpus and catch regressions in already-found crash inputs.
